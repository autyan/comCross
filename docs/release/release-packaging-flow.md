# Release Packaging Flow

This document captures the staged release pipeline used by the
`feature/release-packaging-flow` work.

## Goal

ComCross releases should be generated by automation and should publish legal
OS-level packages:

- Windows MSI
- Linux DEB/RPM
- Linux AppImage fallback
- Checksums and signatures

Portable `.tar.gz` and `.zip` archives are no longer the default public release
format.

## Stages

1. Build publish output with `scripts/release/build-publish-output.sh`.
2. Build Linux DEB/RPM packages with `scripts/release/build-linux-packages.sh`.
3. Build Linux AppImage packages with `scripts/release/build-appimages.sh`.
4. Build Windows MSI packages with `scripts/release/build-windows-msi.ps1`.
5. Generate checksums with `scripts/release/generate-checksums.sh`.
6. Optionally sign checksums with `scripts/release/sign-artifacts.sh`.
7. After local and Windows validation, add the GitHub Actions release workflow.

## Local Linux Verification

```bash
scripts/release/local-verify.sh --version 0.5.0
```

This requires:

- .NET SDK
- Docker
- curl

The Linux DEB/RPM step uses Docker to host `fpm`.

## Windows MSI Verification

Run on Windows:

```powershell
dotnet tool install --global wix
scripts/release/build-windows-msi.ps1 -Version 0.5.0
```

Signing is optional for local smoke tests, but formal release jobs must require
signing material.

## Signing Material

Local verification accepts file paths:

```bash
scripts/release/local-verify.sh \
  --version 0.5.0 \
  --gpg-private-key ~/.keys/comcross-release.asc
```

GitHub Actions must use secrets and temporary files. Signing keys must never be
committed to the repository.
