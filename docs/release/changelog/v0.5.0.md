# v0.5.0 Changelog

## Release Goal

Establish the first automated signed release packaging flow for ComCross and
define the official desktop package support baseline.

## User-Facing Changes

- Official Windows MSI packages for x64 and ARM64.
- Official Linux DEB packages for Ubuntu and Debian on x64 and ARM64.
- Official Linux RPM packages for Fedora on x64 and ARM64.
- Linux AppImage packages as fallback packages for non-mainstream
  distributions.
- Supported operating system baseline documented in the root README and the
  packaging guide.
- Portable `.tar.gz` and `.zip` archives are no longer default GitHub Release
  assets.

## Release Engineering

- GitHub Actions release workflow for manually triggered release builds.
- Signed release workflow validation with `require_signing=true`.
- Windows MSI signing support through GitHub Actions secrets.
- GPG-signed `SHA256SUMS` support for formal releases.
- Official plugin package signing during release package generation.
- Unified final release package directory and checksum generation.
- Release workflow support for `release_notes_path`.
- Release build output validated with zero compiler/analyzer warnings after
  async boundary cleanup.

## Compatibility

- The v0.4.0+ workspace format remains incompatible with old v0.3 development
  state. Existing sessions from older development builds should be recreated.
- Linux DEB/RPM packages are framework-dependent and require the .NET 8 runtime.
- Linux AppImage packages are self-contained.
- Windows MSI packages are self-contained and install per user.
- macOS packages are not provided.
- Non-mainstream Linux distributions are outside the formal support commitment.

## Documentation

- Added release notes source files under `docs/release/notes/`.
- Added version changelog files under `docs/release/changelog/`.
- Updated the packaging guide with supported operating systems and release
  workflow usage.
- Updated release packaging flow documentation with GitHub Actions inputs and
  final release trigger examples.
- Added AI collaboration rules for async boundary decisions.

## Known Limitations

- The Windows signing certificate used during current validation is a
  self-signed test code-signing certificate, not a publicly trusted commercial
  certificate.
- Full Linux distribution matrix testing is deferred until user volume justifies
  maintaining dedicated package validation environments.
- AppImage packages are fallback packages, not a guarantee that every Linux
  desktop environment is fully validated.
- GitHub Actions currently uses manually triggered release jobs rather than a
  tag-push automatic release.
