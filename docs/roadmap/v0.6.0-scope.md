# ComCross v0.6.0 Scope

## Theme

ComCross v0.6.0 focuses on data interpretation and high-volume message
experience. After v0.4.0 stabilized the runtime architecture and v0.5.0
stabilized release engineering, v0.6.0 should make captured communication data
easier to parse, search, and analyze.

## Primary Goals

- Publish the first official ComCross website as the public project, download,
  and documentation entry point.
- Establish the first supported custom protocol profile model.
- Rework the existing CPDL prototype into a bounded declarative protocol
  description system.
- Add protocol analyzer execution and derived parse results without changing
  transport plugin responsibilities.
- Improve message storage, search, and large-message-list rendering so the UI
  does not depend on keeping all messages in memory.
- Add the first official data analysis consumer plugin based on standardized
  messages, attributes, and decoded fields.

## Architecture Principles

- Bus plugins remain producers of transport facts: raw messages, message
  attributes, session metadata, reconnect policy, topology, and plugin-owned
  connection semantics.
- Protocol parsing is a consumer-side capability. TCP, UDP, and Serial plugins
  must not parse user protocols or own protocol-specific business meaning.
- Raw messages remain the source of truth. Parsed data is derived data and must
  be rebuildable from raw message data and the selected protocol profile.
- Protocol parsing must be bounded by frame size, field count, execution time,
  warning count, and failure isolation.
- Analysis plugins consume standardized messages and derived facts from the main
  program. They must not infer private bus plugin semantics.

## Scope 0: Official Website And Public Project Entry

ComCross should have a small official website before the v0.6.0 feature work
starts to broaden. The first version should be a project and download portal,
not a separate documentation authority or marketing-heavy product site.

Required capabilities:

- Static website source under `website/` that can be deployed without adding a
  JavaScript build toolchain.
- Product positioning, supported platforms, download entry, GitHub entry, and
  documentation entry.
- A visual product preview that reflects the current Shell workflow without
  promising unfinished v0.6.0 behavior.
- Clear v0.6.0 preview content that uses planned language for protocol
  profiles, analyzer pipeline, large-message handling, and analysis views.
- GitHub Pages hosting path documented for the first public deployment.

Out of scope for the first website version:

- Donation, sponsorship, or monetization entry points.
- Blog, news system, or full documentation-site migration.
- Full website localization.
- Claims that exceed the current English README, release notes, changelog, or
  specifications.

## Scope 1: Custom Protocol Profile And CPDL

The current CPDL prototype under `src/Core/Cpdl` should be evolved, not
discarded. It already provides a useful foundation: a declarative DSL, lexer,
parser, AST, semantic analysis, code generation, sample `.cpdl` files, and
tests.

v0.6.0 should turn CPDL into the first version of Custom Protocol Profile:

- Profile metadata: id, name, version, description, author, and compatibility
  version.
- Frame rules:
  - delimiter-based
  - fixed-length
  - length-field-based
  - optional header/tail match
  - maximum frame length
- Message matching and dispatch:
  - match by fixed bytes, field values, length, or simple expressions
  - support multiple message definitions per profile
  - return a structured unknown-message result instead of throwing into the UI
- Field decoding:
  - numeric integer types
  - float types where byte order is explicit
  - fixed-length bytes and strings
  - endian handling
  - scale/unit metadata
  - enum mapping
- Validation:
  - range checks
  - required length
  - magic/header checks
  - bounded checksum support for common algorithms
  - structured parse warnings and failure reasons
- Profile lifecycle:
  - create, edit, duplicate, import, export
  - bind/unbind profile to a session
  - enable/disable parsing per session
  - test sample input and preview parse result

Out of scope for v0.6.0:

- General scripting language execution.
- Arbitrary user code.
- Automatic protocol inference.
- Complex state-machine protocols.
- Multi-packet reassembly beyond bounded frame splitting.
- Stable third-party plugin SDK promises for protocol extensions.

## Scope 2: Protocol Analyzer Pipeline

Protocol analysis should consume stored raw messages and selected protocol
profiles, then produce derived parse results.

Required model:

- Input: message id, session id, timestamp, direction, payload bytes, and
  message attributes.
- Output: profile id, message type, decoded fields, parse status, warnings,
  derived attributes, and raw-frame offsets.
- Parse failures must be visible, searchable, and isolated from message
  receiving/sending.
- Analyzer execution should be cancellable and should not block transport IO or
  Shell rendering.
- Parsed results should be cacheable or persistable enough for search and
  analysis, while preserving raw messages as the authoritative source.

The old `ProtocolMessageStreamService` direction should not become the primary
v0.6.0 model. It can inform the design, but v0.6.0 should integrate with the
current message store/search pipeline instead of creating a separate physical
frame list.

## Scope 3: Message Store, Search, And Large Data Display

v0.6.0 should reduce memory pressure from large message sessions and make
message search a first-class operation.

Required capabilities:

- Append-first message storage model that can persist messages promptly.
- Windowed or virtualized message list loading.
- Search tasks with cancellation, progress, and result navigation.
- Search over:
  - payload text or hex representation
  - direction
  - timestamp range
  - message attributes
  - decoded fields and parse status
- Session deletion remains destructive: deleting a session deletes its messages,
  derived parse results, and plugin-owned session data.

Out of scope for v0.6.0:

- Adaptive storage engines selected by disk benchmark.
- Cross-workspace full-text search.
- Cloud sync.
- Long-term archival/reporting workflows.

## Scope 4: Official Data Analysis Consumer

The official analysis capability should be a consumer of standardized message
and derived data.

Initial views:

- RX/TX throughput over time.
- Message count timeline.
- Payload size distribution.
- Parse warning/error counts.
- Attribute and decoded-field aggregation.
- Simple line chart from one selected numeric decoded field.

Filtering should support session, direction, attributes, decoded fields, and
parse status.

Out of scope for v0.6.0:

- Full BI/dashboard builder.
- User script execution.
- Multi-session correlation engine.
- Persistent report scheduling.

## Release Acceptance

v0.6.0 should be considered complete only when:

- A user can define or import a bounded CPDL profile and bind it to a session.
- Raw messages remain visible and unchanged.
- Parsed results are visible as derived data and can be searched.
- Large message sessions do not require full message history in a Shell
  collection.
- The official analysis consumer can render useful basic charts from messages
  and decoded fields.
- TCP, UDP, and Serial plugins do not change their transport ownership model to
  support protocol parsing.
