# ComCross v0.6.0 Scope

## Theme

ComCross v0.6.0 focuses on data interpretation and high-volume message
experience. After v0.4.0 stabilized the runtime architecture and v0.5.0
stabilized release engineering, v0.6.0 should make captured communication data
easier to parse, search, and analyze.

## Primary Goals

- Publish the first official ComCross website as the public project, download,
  and documentation entry point.
- Establish file-backed Session Logs as the default high-volume message
  workflow and add optional session-level archive storage.
- Improve message storage, search, and large-message-list rendering so the UI
  does not depend on keeping all messages in memory.
- Provide basic export for complete available session data from live spool or
  archive history mode.
- Keep custom protocol profile and decoder work as a later stage after the
  message storage foundation is stable.

## Architecture Principles

- Bus plugins remain producers of transport facts: raw messages, message
  attributes, session metadata, reconnect policy, topology, and plugin-owned
  connection semantics.
- Session Logs are communication logs backed by a bounded file spool. They are
  not Application Logs and are not a long-term retention promise.
- Session Archive is a user-enabled, session-level SQLite store for long-term
  history browsing. It is not enabled by default and must not change the spool
  structure or cleanup behavior.
- Message storage pressure and data loss must be observable. Silent loss is not
  acceptable.
- Decoder, protocol profile, and derived-data features must build on the stable
  message frame model rather than changing transport plugin responsibilities.

## High-Volume Product Positioning

ComCross should keep high-volume communication capture as part of the product
direction, but the v1.0 target is not a general-purpose line-rate packet capture
engine.

The intended advantage is storage-first long-running communication work:

- embedded, industrial, automotive, and device communication sessions;
- multi-source communication logs that need stable session identity;
- long-running capture where memory-first or UI-first tools become unreliable;
- visible overload, cleanup, and loss instead of silent failure;
- export, replay, analysis, and future decoder workflows based on landed
  frame facts.

This positioning allows ComCross to replace part of the Wireshark/tcpdump
workflow for device communication and high-throughput logs, while avoiding a
premature commitment to kernel-level capture, packet-driver integration,
pcapng-first storage, or line-rate gigabit capture before the storage-first
model is proven.

The practical throughput boundaries are:

- v1.0 core target: up to roughly 32 MB/s sustained communication data with
  usable browsing, export, and visible degradation;
- v1.x professional target: 32-100 MB/s sustained, with landing prioritized and
  viewer/archive/analysis allowed to degrade;
- future capture-engine target: above 100 MB/s sustained or true line-rate
  network capture, handled as a dedicated capability line after field
  validation.

## Scope 0: Official Website And Public Project Entry

ComCross should have a small official website before the v0.6.0 feature work
starts to broaden. The first version should be a project and download portal,
not a separate documentation authority or marketing-heavy product site.

Required capabilities:

- Static website source under `website/` that can be deployed without adding a
  JavaScript build toolchain.
- Product positioning, supported platforms, download entry, GitHub entry, and
  documentation entry.
- A visual product preview that reflects the current Shell workflow without
  promising unfinished v0.6.0 behavior.
- Clear v0.6.0 preview content that uses planned language for Session Logs,
  archive history mode, large-message handling, and analysis views.
- GitHub Pages hosting path documented for the first public deployment.

Out of scope for the first website version:

- Donation, sponsorship, or monetization entry points.
- Blog, news system, or full documentation-site migration.
- Full website localization.
- Claims that exceed the current English README, release notes, changelog, or
  specifications.

## Scope 1: Session Logs, Spool, And Archive

v0.6.0 should first establish the message storage foundation documented in
`docs/specs/13-Session-Logs-And-Archive-Design.md` and accepted through
`docs/roadmap/v0.6.0-session-storage-acceptance.md`.

Required capabilities:

- File-backed Session Logs as the default live message store.
- Global and per-session spool capacity limits.
- Observable spool cleanup and evicted-history states.
- Per-session SQLite Session Archive that is disabled by default and requires
  manual user confirmation.
- Archive history data mode that lets the message viewer read from archive.
- Storage calibration, hardware/storage fingerprint, conservative
  fallback, and visible overload/loss reporting.
- Message viewer data source separation: live spool vs archive history.
- Message viewer display modes: Plain, Slim, and Detailed.
- Basic export of complete available data from live spool or archive.
- Export modes for Plain, Slim, and Detailed text output, including attributes
  in Detailed output.

Out of scope for v0.6.0:

- Archive replay.
- Archive import, migration, backfill, or synchronization.
- Portable archive package export.
- Partial range export.
- Protocol decoding and CPDL integration.
- Decoded-field persistence or search.
- Payload full-text indexing.

## Scope 2: Message Store, Search, And Large Data Display

v0.6.0 should reduce memory pressure from large message sessions and make
stored message browsing/search a first-class operation.

Required capabilities:

- Windowed or virtualized message list loading from live spool.
- History data mode loading from session archive when archive is enabled.
- Search tasks with cancellation, progress, and result navigation.
- Search over:
  - payload text or hex representation from available message data
  - direction
  - timestamp range
  - message attributes
- Session deletion remains destructive: deleting a session deletes its spool,
  archive when selected in the confirmation, and plugin-owned session data.

Out of scope for v0.6.0:

- Decoded-field search.
- Protocol parse status search.
- Cross-workspace full-text search.
- Cloud sync.
- Portable archive import/export.
- Partial range export.

## Scope 3: Lightweight Analysis Consumer

The official analysis capability should consume standardized frames from the
live spool first. It should remain a consumer of Core-provided frame data and
must not infer private bus plugin semantics.

Initial views:

- RX/TX throughput over time.
- Message count timeline.
- Payload size distribution.
- Attribute aggregation where attributes are available.

Filtering should support session, direction, timestamp range, and attributes.

Out of scope for v0.6.0:

- Decoded-field aggregation.
- Parse warning/error counts from protocol decoding.
- Full BI/dashboard builder.
- User script execution.
- Multi-session correlation engine.
- Persistent report scheduling.

## Scope 4: Decoder And Protocol Profile Design Holding Area

Custom protocol, CPDL, and decoder work remain important, but v0.6.0 should not
commit to the full implementation before the message storage foundation is
stable.

Confirmed direction:

- `Decode Me` should become a message-frame-level entry point in a later scope.
- Decoder definitions may become user assets.
- Decode results are not part of the v0.6.0 storage commitment.
- Full stream frame splitting, source partitioning, CPDL profile lifecycle,
  parsed-result persistence, and decoded-field search are deferred.

Out of scope for v0.6.0:

- General scripting language execution.
- Arbitrary user code.
- Automatic protocol inference.
- Complex state-machine protocols.
- Stable third-party plugin SDK promises for protocol extensions.

## Release Acceptance

v0.6.0 should be considered complete only when:

- Session Logs are backed by bounded file spool storage by default.
- Session Archive can be enabled manually for a session and queried through
  history data mode.
- Large message sessions do not require full message history in a Shell
  collection.
- Storage calibration, conservative fallback, and visible overload/loss reporting
  are in place.
- Plain, Slim, and Detailed message viewer modes are available.
- Complete available data can be exported from live spool or archive history
  mode.
- Raw frames remain visible and unchanged.
- TCP, UDP, and Serial plugins do not change their transport ownership model to
  support message storage, analysis, or future decoding.
