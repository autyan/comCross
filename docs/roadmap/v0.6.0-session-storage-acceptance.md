# v0.6.0 Session Logs And Archive Acceptance

This document defines the acceptance criteria for the first v0.6.0 feature
scope: file-backed session logs, optional session archive storage, and the
message viewer foundations required before decoder or analyzer work resumes.

## Prerelease Result

The v0.6.0 prerelease branch was accepted around the archive and message-viewer
foundation. Archive remains optional and session-scoped, live/archive viewer
data-source separation is in place, and Plain, Slim, and Detailed display modes
are available.

The original acceptance list below remains the detailed checklist used during
implementation review. Items that are intentionally not part of the accepted
v0.6.0 scope are documented as deferred in:

```text
docs/release/changelog/v0.6.0.md
docs/release/prerelease/v0.6.0-review.md
docs/roadmap/v0.6.1-scope.md
```

## Scope

In scope:

- file-backed session spool as the default Session Logs implementation;
- bounded global and per-session spool cleanup;
- optional per-session SQLite archive;
- archive history data mode for message browsing;
- basic export of complete available data from live spool or archive;
- storage calibration and conservative fallback policy;
- storage backpressure and visible data-loss/error reporting;
- message viewer data-source and display-mode model;
- user-facing settings and confirmations for Session Logs and Session Archive.

Out of scope:

- protocol decoding;
- CPDL profile implementation;
- archive replay;
- archive import, migration, or backfill;
- portable archive package export;
- partial range export;
- archive analysis features;
- decoded result persistence;
- payload full-text indexing.

## Acceptance Criteria

### 1. User Model

- [ ] Settings and UI distinguish `Application Logs`, `Session Logs`, and
      `Session Archive`.
- [ ] Settings expose Session Logs storage status and basic configuration.
- [ ] Per-session UI exposes Archive management entry points.
- [ ] Session Logs are described as bounded communication logs, not guaranteed
      long-term retention.
- [ ] Session Archive is described as an explicit session-level long-term
      storage choice.
- [ ] Archive enablement requires manual user action and a second confirmation.
- [ ] There is no global setting that enables archive by default for all
      sessions.

### 2. Session Spool

- [ ] RX/TX frames are written to a file-backed session spool by default.
- [ ] The spool uses stable session IDs for storage identity, not session names.
- [ ] Frame IDs are assigned once by Core ingest and reused across all storage
      consumers.
- [ ] The spool supports global maximum size and per-session maximum size.
- [ ] Cleanup does not remove the active write target.
- [ ] Cleanup updates available-history metadata.
- [ ] Requests for cleaned history produce an evicted-history state.
- [ ] Cleanup affecting visible current-session history is user-visible.
- [ ] The implementation does not require all historical frames to remain in
      Core or Shell memory.

### 3. Session Archive

- [ ] Archive is enabled per session only.
- [ ] Enabling archive does not change spool file structure or spool cleanup.
- [ ] Archive stores only new frames after enablement.
- [ ] Archive does not backfill existing spool data in v0.6.0.
- [ ] Disabling archive stops future writes but keeps existing archive data.
- [ ] Archive deletion is a separate confirmed operation.
- [ ] Archive write failure does not fail spool writes.
- [ ] Archive write failure notifications are throttled and not emitted per
      frame.

### 4. History Data Mode

- [ ] Message viewer can read from live spool mode.
- [ ] Message viewer can read from archive history mode when archive exists.
- [ ] Live and archive reads expose the same structured frame model.
- [ ] Archive history mode supports basic window queries by frame ID and latest
      range.
- [ ] Archive history mode does not expose replay or analysis-only features in
      this scope.
- [ ] Archive history mode can browse archived frames after corresponding spool
      data has been cleaned.

### 5. Basic Export

- [ ] Export can write the complete available data from `LiveSpool`.
- [ ] Export can write the complete available data from `Archive` when archive
      exists.
- [ ] Export writes plain-text external files and does not expose spool segment
      files as the user-facing export format.
- [ ] Export supports Plain, Slim, and Detailed text modes.
- [ ] Detailed export includes frame ID, timestamp, direction, source, raw data,
      and attributes.
- [ ] Export does not require all exported frames to be loaded into Shell memory
      at once.
- [ ] Export output is deterministic and safe for binary payloads.
- [ ] Export communicates partial failure or source data loss to the user.
- [ ] Export does not expose archive import, portable archive package export,
      replay package export, or protocol-decoded result export in this scope.

### 6. Message Viewer Display Modes

- [ ] `Plain` mode renders raw data only.
- [ ] `Slim` mode renders an RX/TX marker plus raw data.
- [ ] `Detailed` mode remains available for metadata, attributes, and future
      actions.
- [ ] Plain and Slim modes do not render cards or extra vertical spacing between
      frames.
- [ ] Plain and Slim copy behavior is close to rendered text, with one newline
      between copied frames and no extra blank lines.

### 7. Calibration And Backpressure

- [ ] Storage uses a conservative policy before calibration completion.
- [ ] Storage calibration estimates spool write/read capacity.
- [ ] Archive write/query capacity is calibrated before or when archive is
      enabled, or archive uses conservative policy until measured.
- [ ] Hardware/storage fingerprint changes reset storage policy to conservative.
- [ ] Calibration completion can switch storage policy automatically.
- [ ] Calibration does not expose a benchmark platform, score report, or tuning
      suite as part of the MVP.
- [ ] Storage overload, frame loss, cleanup loss, and archive failure are
      visible through logs and user notifications.
- [ ] Storage backpressure integrates with existing bus pressure handling.

### 8. Deletion And Abnormal Data

- [ ] Deleting a session deletes its spool data.
- [ ] If archive exists, session deletion confirms archive deletion and selects
      archive deletion by default if archive deletion is presented as a separate
      option.
- [ ] Core detects and reports spool metadata read failure, spool data read
      failure, cleanup failure, archive open failure, unsupported archive
      schema, and session/archive identity mismatch.
- [ ] Recoverable issues are repaired or isolated with diagnostics.
- [ ] Non-recoverable issues do not silently break startup or message browsing.

```bash
bash repo-tools/check-shell-i18n.sh
```
